1. LDAP — Lightweight Directory Access Protocol

LDAP is the language used to query Active Directory. Active Directory stores objects (users, computers, groups, OUs) in a hierarchical database; LDAP is the protocol you speak to read or modify them. When BloodHound, PowerView, or ldapsearch ask AD a question, they speak LDAP under the hood.

Key facts

  • Port 389 (LDAP) — cleartext by default
  • Port 636 (LDAPS) — TLS-wrapped
  • Port 3268 / 3269 — Global Catalog (forest-wide LDAP)
  • Hierarchical (tree-structured) — every object has a path
  • Read-heavy: optimised for reading the directory, not transactional writes
  • Authentication is called a bind: anonymous, simple (user+pass), or SASL (Kerberos/NTLM)

Why an attacker cares

  • Anonymous bind sometimes leaks the entire directory
  • Any authenticated user can read most of AD by default
  • Maps users → groups → ACLs — BloodHound is just LDAP queries + graph theory
  • LDAP signing off + relay = remote DC compromise
  • Reveals servicePrincipalName for Kerberoasting

2. Distinguished Names (DN) — How AD Addresses Things

Every object in AD has a unique path called a Distinguished Name. Read it right to left — like a postal address that starts specific and gets broader.

CN=adunn , OU=IT Admins , OU=Corp , DC=inlanefreight , DC=local

Hover any segment. Read right→left: domain → OUs → object.

Common attribute prefixes

CN

Common Name
The actual object — a user, group, computer, or container. Example: CN=adunn.

OU

Organizational Unit
A folder that holds objects and applies policy. Example: OU=IT Admins.

DC

Domain Component
One label of the domain DNS name. inlanefreight.local → DC=inlanefreight,DC=local.
Don't confuse: DC= in a Distinguished Name means Domain Component — it does not mean Domain Controller. Same letters, totally different concept.

Filters use prefix notation: (operator attribute=value). Combine with & (AND), | (OR), ! (NOT). Hover each example for a breakdown.

6. AD Building Blocks

Click any card to expand. Each block is one piece of the AD puzzle.

🌲

Forest

≈ The property line / fence
The whole environment — and the only real security boundary in AD. Contains one or more domain trees that share schema + global catalog.
Owns: Schema partition, Configuration partition, forest-wide FSMO roles (Schema Master, Domain Naming Master), Enterprise Admins, Schema Admins.

Why it matters: Cross-domain trust within a forest is automatic and transitive. SID Filtering is OFF by default intra-forest, so owning a child domain often = owning the forest (ExtraSids attack).
🏢

Domain

≈ A floor with its own keycards
One administrative unit inside a forest. Has its own users, groups, password policy, and DCs.
Owns: Domain naming context (the user/group/computer DB), domain-wide FSMO roles (PDC Emulator, RID, Infrastructure), its own KRBTGT.

Why it matters: Most attacks target a single domain at a time. Compromising one domain != compromising the forest, but in practice it usually leads there.
📂

OU — Organizational Unit

≈ Office suite with house rules
A folder inside a domain for grouping users/computers and applying GPOs. Can be nested.
Receives: GPOs (inherited down the tree), delegated administrative rights.

Why it matters: An attacker with rights on an OU (GenericWrite, WriteDACL) can link a malicious GPO and own everything inside. Also: OU=Domain Controllers is the most sensitive container in AD.
📜

GPO — Group Policy Object

≈ The house rules themselves
A bundle of settings (registry keys, scripts, software, security policy) linked to a Site, Domain, or OU and pushed to every computer/user underneath.
Stored in: SYSVOL share + LDAP. Computers re-read every 90 ± 30 min.

Why it matters: A GPO can deploy a scheduled task to run as SYSTEM on every host below it. Misconfigured permissions on SYSVOL or AD GPO objects = mass code execution. Old GPP files (Groups.xml) leak cpassword — AES-decryptable cleartext local admin creds.
🖥

DC — Domain Controller

≈ The master keychain cabinet
The server that runs AD. Stores ntds.dit (every user's NT hash) and runs Kerberos KDC, LDAP, DNS, and the Global Catalog.
Listens on: 88 (Kerberos), 389/636 (LDAP/S), 3268/3269 (GC), 53 (DNS), 445 (SMB), 135 + dynamic (RPC).

Why it matters: Owning the DC = owning the domain. DCSync remotely abuses replication to pull every credential without ever logging into the DC itself. secretsdump.py -just-dc is the finishing move.
🔑

KRBTGT

≈ The owner's invisible-ink pen
A hidden, disabled account whose NT hash signs every Kerberos ticket in the domain. Created automatically when the domain is built.
Stored in: ntds.dit on the DC.

Why it matters: If you steal the KRBTGT NT hash, you can forge a Golden Ticket — a valid TGT for any user (including non-existent ones) for up to 10 years. Defense: rotate the password TWICE (each rotation invalidates one history slot, and tickets verify against current + previous).
Rule of thumb: The forest — not the domain — is the real security boundary. Cross-domain trust inside a forest is mostly automatic and transitive.
