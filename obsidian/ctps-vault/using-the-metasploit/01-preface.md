## Description
Framing for why automated frameworks like Metasploit matter — time pressure, discipline, and tunnel vision risks for the penetration tester.

## Tags
metasploit, philosophy, methodology, discipline

## TL;DR — What's Important
- Tools save time but create comfort zones and tunnel vision — don't let "the tool can't do it" become "neither can I."
- You're not racing other infosec people; you're racing the clock and the customer's expectations.
- Read the docs for every tool you use — unpredictable behavior on a customer environment is the fastest route to a lawsuit.
- Validate vulnerabilities, not your ego.

## What This Section Covers
The Preface argues against both extremes — purists who refuse all automation and beginners who let the tool think for them. A pentest has finite time and the customer cares about coverage, not artistry. Use tools as time-savers for the routine work so brainpower goes to the hard parts.

## Key Concepts
- **Tools are time-savers, not a backbone** — never depend on them to define what an assessment can or can't reach.
- **Audit the tool itself** — some tools leave artifacts on the target (files, listeners, registry entries) that create liability or get you caught.
- **Methodology beats tooling** — a solid methodology + preliminary checks framework means tools accelerate good process instead of substituting for it.

## Key Takeaways
- Time is the binding constraint on every real engagement — automation exists so the hard hours go where they're needed.
- Tools that you don't understand are tools that can burn you and the customer simultaneously.
- "Tunnel vision" is the named failure mode of over-relying on automation — explicitly fight it by trying things the tool doesn't suggest.
